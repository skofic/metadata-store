⏺ The idea is appealing at the surface — _type is cleaner to write and read — but it runs into a structural constraint that's baked into
  this dictionary's design.
                                                                                                                                           
  The invariant it breaks: property keys in _type_object documents are expected to be term _gids. That invariant is what makes the schema  
  self-describing and navigable. If you write "_type": "_type_number_float" inside a _scalar object, the key _type has to resolve to a term
   — but which one? _scalar_type and _comparable_type are different terms with different _data sections and different allowed value sets.  
  You'd need a contextual lookup ("this _type is inside _scalar, so it means _scalar_type") that adds a layer of indirection the validator
  doesn't currently have.

  The GID collision: _type is already taken — it's the root namespace term for the entire type enumeration (_nid: "", _lid: "type", _gid:  
  "_type"). Using it also as a property descriptor creates a collision with its own namespace.
                                                                                                                                           
  The subtle irony: the current naming already does exactly what you're describing. _scalar_type has _nid: "_scalar" and _lid: "type" — so 
  the namespace already encodes "the type property within the _scalar context." The human-readable JSON key is longer, but the conceptual
  model is identical to your proposal.                                                                                                     
              
  Where it could work: if this dictionary eventually gets a rendering/serialisation layer (a Foxx service or API), that layer could use    
  short aliases — emit _type in JSON output and map it back to _scalar_type/_comparable_type/etc. based on the enclosing context during
  ingestion. The internal representation stays unambiguous; the external API gets the ergonomic names. That's a Phase 6/7 concern.         
              
  For now, the cost of the current verbose names is low and the benefit of unambiguous self-description is high. I'd keep the status quo.

⏺ It's a coherent generalization. You're pushing _path_data from carrying only rules to carrying the full term-card payload — both _data
  (what type the property has in this context) and _rule (what companions are permitted). One _type term; its meaning is fully determined
  by the edge.                                                                                                                             
   
  What it buys:                                                                                                                            
  - Eliminates _scalar_type, _comparable_type, _dict_key_type as separate terms — they collapse into a single _type with context-specific
  definitions in edges                                                                                                                     
  - The edge becomes the single source of truth for both the type constraint and the structural rules of a property in a given context
  - Scales cleanly across levels — _type inside _comparable inside _set inside _array is a distinct path root, so a distinct edge, so a    
  distinct definition                                                                                                                      
                                                                                                                                           
  What it costs:                                                                                                                           
  - The dictionary's self-description becomes more distributed — you can no longer read a term card to understand a property; you have to  
  traverse to its edges per context                                                                                                        
  - The validator needs edges not just for rule evaluation but also for type resolution — it becomes fully edge-dependent
  - _path_data goes from a thin annotation layer to embedding mini term-cards, which blurs the boundary between terms and edges            
                                                                                                                                           
  The honest verdict: it's worth following, but at Phase 6, not now. The right moment is when you're actually building the validator —     
  because the cost of the added complexity only becomes concrete when you're writing the traversal code. At that point you'll know whether 
  the reduction in term proliferation justifies the distributed self-description. Designing it in the abstract risks committing to a       
  tradeoff you can't fully evaluate yet.                                                                                                   
              
  Worth capturing as a design note somewhere.




Embedded validation logic and ArangoDB UDFs (2026-04-08):

Two ideas explored for extending validation beyond what the declarative _data section can express.

Idea 1 — embed validation code directly in _data section properties (e.g. a new _code type whose value is a JavaScript function). The validator would load and execute it against the current value. The security problem is obvious, but the deeper issue is structural: the dictionary is a declarative system — _data describes what data should look like, and a separate interpreter applies that description. Embedding executable code collapses that separation. The dictionary becomes self-executing rather than self-describing. Validation rules can no longer be audited by reading the data; you have to read and trust code stored in the database. Not a path worth taking.

Idea 2 — ArangoDB User-Defined Functions (UDFs). ArangoDB allows registering JavaScript functions via the HTTP API (stored in _aqlfunctions), callable in AQL as NAMESPACE::FUNCTION(args). Foxx services can call them through AQL. The critical constraint: UDFs are pure functions with no database access from inside the function body — no db._query(), no collection lookups, no HTTP. This matters because: (a) format checks (email, regexp, IPv4) are exactly what _regexp and the type system already express declaratively; (b) semantic checks (is this _gid a real term? is this value a member of this enumeration?) require database access and must stay in Foxx/AQL anyway. UDFs handle the easy half the type system already covers, and can't touch the hard half. They also live outside the dictionary data, breaking the self-contained property and requiring a separate deployment step.

Verdict: the typedef system is the dictionary-native extensibility mechanism. New types are introduced by creating a typedef term (structural definition in _data) plus a handler case in the Foxx validator (implementation). The dictionary stays declarative; code stays in code, versioned and testable. Both embedded-code ideas are not improvements on this.
