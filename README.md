# Metadata Store

**A database-hosted metadata repository and data dictionary**

---

## Overview

This repository defines a self-describing [data dictionary](https://en.wikipedia.org/wiki/Data_dictionary) and metadata repository hosted in an [ArangoDB](https://www.arangodb.com/) NoSQL graph database. It provides a formal, structured vocabulary for describing datasets, variables, and data standards — including their types, units, valid ranges, controlled vocabularies, and relationships.

The dictionary is **self-describing and recursive**: the structures used to define terms are themselves terms in the dictionary. This makes the ontology navigable and machine-readable without external schema files.

## Repository Layout

### `data/core/` — Core dictionary terms

The foundational terms that define the dictionary's own structure. All are in the default namespace (`_nid: ""`), giving their `_gid` a leading underscore.

| File | Contents |
|---|---|
| `_term.json` | Top-level term structure: `_term`, `_id`, `_key`, `_rev`, `_domn`, `_term_role`, and all `_term_role_*` values |
| `_code.json` | Identifier section: `_code`, `_gid`, `_lid`, `_nid`, `_aid`, `_pid`, `_name`, `_symbol`, `_regexp`, `_emoji` |
| `_info.json` | Documentation section: `_info`, `_title`, `_definition`, `_description`, `_examples`, `_notes`, `_url`, `_citation`, `_provider`, `_methods`, `_usage` |
| `_data.json` | Data shape section: `_data`, `_scalar`, `_object`, `_array`, `_set`, `_nested`, `_tuple`, `_typedef` |
| `_scalar.json` | Scalar type terms and all their companion properties (`_number`, `_string_*`, `_text_*`, `_term_key_*`, `_boolean`, `_enum`, `_timestamp`, `_handle`, range terms, unit terms) |
| `_array.json` | Array and element-count terms: `_array`, `_elements`, `_min-items`, `_max-items` |
| `_dict.json` | Dictionary shape terms: `_dict`, `_dict_key`, `_dict_value` |
| `_object.json` | Object schema terms: `_object`, `_open`, `_closed`, `_required`, `_recommended`, `_banned`, `_computed`, `_locked`, `_immutable`, `_default-value`, `_selector`, `_selectors`, `_selection`, `_all`, `_any` |
| `_edge.json` | Edge document terms: `_edge`, `_from`, `_to`, `_path`, `_path_data` |
| `_predicate.json` | Predicate terms: `_predicate` and all `_predicate_*` values |
| `_type.json` | Built-in typedef terms: `_type`, `_type_scalar`, `_type_comparable`, `_type_key` |
| `_term.enum.json` | Enumeration edges connecting `_term_role_*` terms into the `_term_role` controlled vocabulary |
| `_predicate.enum.json` | Enumeration edges connecting `_predicate_*` terms into the `_predicate` controlled vocabulary |
| `_type.enum.json` | Enumeration edges connecting `_type_*` terms into the `_type` namespace |

### `docs/` — Term cards and reference documentation

Markdown documentation cards generated automatically from the source JSON, one per term. Also contains `docs/reference/DictionaryReference.md` — the hand-authored structural reference — and `docs/images/` with SVG diagrams.

Term cards are generated by the workflow in `workflows/term-cards/`. Never edit cards by hand — edit the source JSON and regenerate.

### `workflows/term-cards/` — Term card generator

A Swift command-line tool that reads the JSON source files in `data/core/` and writes a Markdown card for each term into `docs/`. Run with:

```bash
swift run --package-path workflows/term-cards
```

## Term Structure

Every term is a JSON document with up to five top-level **sections**:

| Section | Required | Purpose |
|---|---|---|
| `_code` | Always | Identifiers: global ID (`_gid`), local ID (`_lid`), namespace (`_nid`), and aliases |
| `_info` | Always (except alias terms) | Multilingual human-readable content: title, definition, description, examples |
| `_data` | Descriptor terms only | Data type and shape (makes a term a *descriptor*) |
| `_domn` | Optional | Categorical classification: term role, subject domain, and other metadata |
| `_prop` | Optional | Concrete attributes of the real-world entity the term represents |

`_gid` is computed as `_nid + "_" + _lid` and doubles as the ArangoDB `_key`. Terms with an empty-string `_nid` are core dictionary building blocks (their `_gid` begins with `_`).

Schema constraints (required/recommended/banned properties) are expressed inline within `_data._object._open` or `_data._object._closed` — not in a separate rule section.

## Data Model Highlights

### Type-as-key convention

In `_scalar`, `_set`, `_nested`, and `_dict_key`, the data type is expressed as a **property key** inside the shape object:

```json
{
    "_data": {
        "_scalar": {
            "_number_float": {
                "_decimals": 2,
                "_unit": "_unit_length_cm"
            }
        }
    }
}
```

The single key present (`_number_float`) identifies the type; its value holds companion properties for that type.

### Built-in typedefs

Three reusable type-list terms define which types are valid in each context:

| Typedef | Used by | Accepts |
|---|---|---|
| `_type_scalar` | `_scalar` | All scalar types |
| `_type_comparable` | `_set`, `_nested` | Comparable types only (excludes text types) |
| `_type_key` | `_dict_key` | String-compatible types only |

### Graph layer

Relationships between terms are **edge documents** in ArangoDB edge collections. Each edge carries:

- `_from` / `_to` — the related nodes (document handles)
- `_predicate` — the relationship type (`_predicate_enum-of`, `_predicate_property-of`, etc.)
- `_path` — the set of graph root handles whose traversal passes through this edge (enables graph sharing without duplication)
- `_path_data` — context-sensitive data attached to the edge, keyed by graph root or node handle

This supports enumeration graphs, bridge graphs (vocabulary subsets), alias resolution, and presence-triggered conditional rules — all within a single unified edge structure.

## Project Roadmap

### Phase 1 — Documentation *(complete)*
Term section design, graph mechanics, and data model documented in `CLAUDE.md`.

### Phase 2 — Core terms *(complete)*
All core dictionary terms defined: 182 terms across 11 source files, each with full `_info`. Enumeration edges for `_term_role`, `_predicate`, and `_type` namespaces complete.

### Phase 3 — Standards *(pending)*
ISO and other international standards as dictionary terms. Primary source: [Debian iso-codes](https://salsa.debian.org/iso-codes-team/iso-codes) (ISO 639, 3166, 4217, 15924). Supplementary: [mledoze/countries](https://github.com/mledoze/countries).

### Phase 4 — Translations *(pending)*
Translations for all `_info` sections, using Debian iso-codes translation files as the baseline.

### Phase 5 — Schema edges *(pending)*
Edge records encoding property-of and value-of relationships for core terms.

### Phase 6 — Validation library *(pending)*
JavaScript/TypeScript library (ArangoDB Foxx or standalone Node.js) that validates values and objects against dictionary rules, including graph-based conditional rules.

### Phase 7 — Management UI *(pending)*
Web application for navigating the ontology, creating and editing terms, and submitting datasets.
